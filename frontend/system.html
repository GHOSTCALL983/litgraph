Overview

LitGraph turns any arXiv paper into an interactive map of its references, similar work, and the authors linking them — and lets you grow that map by expanding any node. It's a small, dependency-light system: a vanilla-JS frontend rendering a Cytoscape.js graph, a pure-Python standard-library server (no web framework), and open scholarly data from the Semantic Scholar Graph API. Everything runs for free with no API key thanks to an on-disk cache.

Design principles: no heavy dependencies (stdlib server; only requests + pypdf), degrade gracefully under rate limits, cache aggressively, keep the whole thing runnable with one python backend/server.py.

Request flow — analyzing a paper

1
You enter an arXiv id (or upload a PDF) and hit Analyze. The browser POSTs to /api/analyze?mode=deep with {"seeds":[…]} (PDFs first go to /api/extract to detect their id/title).
2
server.py routes it to graph_builder.build_graph_multi().
3
For each seed, graph_builder asks sources.py to resolve the paper and fetch its references, similar papers, and authors from Semantic Scholar. Each source is pulled as a large pool, then filtered to arXiv, ranked and trimmed to your limit.
4
Every API response is checked against cache.py first (30-day disk cache) — so repeats are instant and the rate limit is spared. Misses hit the network with throttling + backoff.
5
graph_builder merges everything into one deduplicated set of nodes (papers + authors) and edges (cites / similar / authored), keeping arXiv papers.
6
The JSON graph returns to the browser, which renders it with Cytoscape.js (fcose force layout). Clicking a node → details; clicking Expand POSTs to /api/expand and merges the new neighborhood in place.

Backend modules

server.py

Pure-stdlib ThreadingHTTPServer — no framework. Serves the HTML pages and a small JSON API; parses cookies for admin sessions; sends Set-Cookie on login. Routes /api/analyze, /api/expand, /api/extract, /api/config, and the /api/admin/* family.

graph_builder.py

The core. MODES (quick/deep/complex) set how far to explore. _add_seed builds one paper's neighborhood; build merges many seeds into one graph (dedup by id); expand grows the graph from an existing node. Adds author "authored" edges so co-authors connect.

sources.py

Semantic Scholar Graph API client: resolve, references, similar, author_papers. Polite throttle, exponential backoff, and graceful 429 handling (returns partial + a warning, never crashes). Reads the API key from config.

cache.py

Tiny on-disk JSON cache of every successful API response (SHA-keyed files, 30-day TTL). This is what makes the no-key setup comfortable: repeats are instant and the network is barely touched.

config.py

Persistent config.json: PBKDF2-hashed admin password + salt, the API key, and settings. In-memory sessions, cache/usage stats, and the API-key getter (config → S2_API_KEY env fallback). Honors LITGRAPH_DATA_DIR.

pdf_extract.py

Uses pypdf to pull an arXiv:id or a title out of an uploaded PDF — just enough to resolve the paper on Semantic Scholar (which then provides the real citation graph).

Frontend

A single page, frontend/index.html, built with vanilla JavaScript — no build step, no framework. It renders the graph with Cytoscape.js + the fcose force-layout (loaded from a CDN), and handles: paper search / PDF upload, the batch tray, node detail panel, the Expand action, filter toggles, PNG/JSON export, node search, recent-papers history, the API-key banner, and the guided tour. admin.html and this system.html page are separate self-contained pages in the same style.

Data, cache & rate limits

Source: the Semantic Scholar Graph API — free and open — for references, recommendations, authors, and arXiv ids. Every paper node links straight to arXiv. We keep arXiv papers (open access).

Why not OpenAlex (which needs no key at all)? It can't reliably resolve a paper from its arXiv id — poor arXiv-preprint coverage, no arXiv-id filter — which would break the core "identify the paper you uploaded" step. So Semantic Scholar + a cache is the better free path.

Rate limits: the anonymous pool is limited. LitGraph throttles requests, backs off on 429, caches everything for 30 days, and degrades gracefully (partial graph + a clear message) instead of failing. A free lifelong API key (set in Admin) removes the limits.

Auth & admin

On first run you set an admin password (/api/admin/setup). It's stored as a PBKDF2-HMAC-SHA256 hash with a random salt (200k iterations) — never in plaintext. Login creates a random session token stored in an HttpOnly cookie; admin endpoints check it. From Admin you manage the API key, default depth, cache, and password — and open this page.

Features

  • Interactive graph — papers (sized by citations) + authors, colour-coded, clickable.
  • Per-source controls — set how many references / similar papers / authors to pull, and what counts as most important (most cited / newest / oldest). Presets load starting numbers.
  • Batch / multi-paper — queue several papers (or PDFs); shared references and authors become the connective tissue in one merged graph.
  • Node expansion — click a paper or author and grow the graph outward from it.
  • Guided tour, PNG/JSON export, node search, recent papers.
  • Admin panel — API key, settings, cache/usage stats, password.

How it was built

LitGraph grew feature by feature:

  • Core graph — resolve an arXiv paper on Semantic Scholar, pull references/ similar/authors, render with Cytoscape.
  • Data-source decision — evaluated OpenAlex for a no-key option; rejected it for poor arXiv-id resolution and kept Semantic Scholar.
  • Free-forever hardening — added the disk cache + graceful rate-limit handling so it runs with no key.
  • Product-grade layer — admin panel (auth, API key, settings, stats), first-run setup, guided tour, PNG/JSON export, node search, recent history, API-key banner.
  • Batch / multi-paper mode — merge several papers into one connected graph.
  • Node expansion — progressive, click-to-grow exploration.
  • Zero-build — no bundler, no container; edit a file and restart.

Running it

There's no container and no build step — it's a two-dependency Python app that runs straight from source. config.json (admin password hash + API key) and the response cache live in backend/, or wherever you point LITGRAPH_DATA_DIR. Set PORT to move off 8000.

pip install -r requirements.txt  then  python backend/server.py  →  http://localhost:8000
